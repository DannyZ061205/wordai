wordAI — How It's Built

A PM-focused deep-dive into the real-time collaborative document editor with an integrated AI writing assistant. You will finish knowing every layer: what it is, why it exists, how it works, and what trade-offs were made.

AI1220 · Assignment 2 MBZUAI · 2026 React 18 + FastAPI Tiptap + WebSocket + SSE

Big Picture

wordAI is a browser-based document editor in the spirit of Google Docs, with one defining twist: an AI writing assistant is wired directly into the editor — rewrite, summarise, translate, expand, grammar-check, run a custom instruction, and a Copilot-style ghost-text autocomplete. Multiple people can open the same document and see each other's keystrokes appear within about a tenth of a second, while the backend quietly persists every change to disk.

The product had to solve three distinct engineering problems at once: store and retrieve structured data (documents, users, versions, shares), propagate live edits across clients (real-time collaboration), and stream AI tokens back to the browser (LLM integration). Rather than force one transport to do all three jobs, wordAI uses a different protocol for each: REST/HTTP for CRUD and auth, a WebSocket relay for live edits and presence, and Server-Sent Events for AI streaming. Everything is kept deliberately simple — no Postgres, no CRDT, no Kubernetes — because the brief is a proof-of-concept.

By the end of this guide, you will be able to point at any feature a user sees (the "who's here" bar, the AI panel, the version history, the share-by-link button) and trace it top-to-bottom through React → hooks → API clients → FastAPI routers → service layer → JSON on disk. You will also understand the deliberate compromises (CRDT dropped in favour of a JSON broadcast; PostgreSQL swapped for JSON files) and why each was the right call for a PoC.

⚡ Roadmap

We'll build from the system shape → data at rest → authenticated identity → the document domain → real-time fan-out → AI streaming → rich-text editor + ghost text, culminating in a review of the engineering deviations that define the project's personality.

1
System Architecture & Transport Choices
Map every feature to the right protocol and the right layer
🎯 Motivation

Before wordAI had any working code, the team had to answer one question: what pipe does each feature flow through? A document save (a one-off request) is nothing like a live keystroke broadcast (a continuous stream), which is nothing like an AI token dribbling in a character at a time. Forcing them all through one transport (say, polling every feature over plain HTTP) would either make the UI feel sluggish or waste enormous bandwidth. The project therefore separates concerns by transport.

💡 Intuition

Think of wordAI as a house with three different utility lines: a water line for occasional big deliveries (HTTP for CRUD), a telephone line that stays open all day for small back-and-forth chatter (WebSocket for live edits), and a ticker tape that drops paper slips one at a time (SSE for AI streaming). All three enter the same house (the backend), but each handles the traffic pattern it was built for.

📘 Definition — The system at 30,000 ft

wordAI is a two-sided web application:

  • Frontend: a single-page React 18 app built with Vite, running entirely in the browser at http://localhost:5173. Rich text is edited with Tiptap (a ProseMirror wrapper). State is held in Zustand stores. Styling is Tailwind.
  • Backend: a FastAPI 0.111 server running on http://localhost:8000 under Uvicorn. It exposes REST endpoints, a WebSocket endpoint, and an SSE endpoint. All data is persisted to JSON files on disk under backend/data/.

The two sides communicate over three transports:

  1. HTTP + JWT — auth, document CRUD, version history, sharing, AI history retrieval, per-user AI settings.
  2. WebSocket — live edit broadcast and presence (who's connected to the same document).
  3. Server-Sent Events (SSE) — token-by-token streaming of AI output from the backend.

⚡ Key Insight — the "three pipes" pattern is not arbitrary

Each transport is the minimum-complexity answer to a specific traffic pattern:

  • HTTP for one-shot, request/response shaped work (CRUD). It's stateless, cache-friendly, and every tool on earth understands it.
  • WebSocket for symmetric, many-small-messages-per-second traffic (keystrokes). HTTP would require reconnecting for every frame; long-polling is clumsy.
  • SSE for one-way, server-pushes-many-chunks traffic (LLM streaming). WebSocket would work but is overkill — SSE is a single HTTP response that the server keeps writing to, so it plays nicely with standard load balancers and CORS.

🔢 Example — tracing a single keystroke

Alice types the letter h into a document that Bob also has open. Concretely:

  1. The key event fires Tiptap's update handler in Alice's browser.
  2. useRealtimeSync (frontend hook) throttles for 80 ms then sends a WebSocket frame {"type":"edit","html":"<p>h</p>","origin":"alice-a9f2"}.
  3. The FastAPI ConnectionManager receives the frame and fans it out to every other socket in that document's room — Bob's tab gets it.
  4. Bob's useRealtimeSync sees origin ≠ my clientId, calls editor.commands.setContent(...), and preserves his caret.
  5. Separately, Alice's useAutoSave hook (triggered by the same edit) waits 1.5 s, then sends PATCH /api/documents/{id} over HTTP to persist the content.

The live feel comes from the WebSocket path (~100 ms). Durability comes from the HTTP path (~1.5 s). Neither knows about the other — they are two independent subsystems operating on the same editor state.

⚠️ Common Mistake

Students often assume the WebSocket is the source of truth for the document. It is not. The WebSocket is a pure relay — it never writes to disk. If the backend restarts and no client is connected, the next client to open the doc loads from the JSON file that the HTTP auto-save path wrote. Lose the HTTP path and the document vanishes on restart; lose the WebSocket and collaboration feels laggy but nothing is lost.

EasyQ: If the browser needs to receive many small AI tokens as they're generated, which of the three transports is chosen, and why?

SSE. The flow is strictly one-way (server → client), so a full bidirectional WebSocket is overkill. SSE rides on a normal HTTP response, so it inherits CORS, auth headers, and proxy behaviour with no extra work.

MediumQ: Why does the auto-save still run even though the WebSocket is already broadcasting every keystroke to everyone?

The WebSocket is ephemeral — it forwards frames but never writes to disk. If no one had run HTTP auto-save, every document would evaporate the moment the last client disconnected. The two paths have different responsibilities: WebSocket = live fan-out; HTTP = durability.

We just said the HTTP path is what makes data durable. So where does that data actually live? Not in PostgreSQL — in plain JSON files. Unit 2 explains how that works without becoming a threading nightmare.

2
JSON File Storage Layer
A single Python dict in memory, flushed to disk on every write
🎯 Motivation

Assignment 1 called for PostgreSQL. Running Postgres means installing it, configuring it, writing migrations, and maintaining connection pools — a lot of ceremony for a grader to set up before they can try your PoC. The assignment explicitly allows file-based persistence, and the team took that escape hatch. But "file-based" still has to answer: how do we read it fast, and how do we stop two threads from corrupting the file at the same time?

💡 Intuition

Imagine a library where the full catalogue lives on a whiteboard on the wall (fast to read), and every time a librarian adds a book they copy the whole whiteboard into a ledger locked in a safe (durable). Only one librarian holds the key to the safe at a time (a single lock). No database. No migrations. Just a dict in memory and a JSON file on disk.

📘 Definition — JSONStore

In backend/app/storage/json_store.py, the JSONStore class holds seven Python dicts — one per entity type — and a threading.Lock. On process startup, each dict is hydrated from its corresponding JSON file. Every mutating method acquires the lock, updates the in-memory dict, and calls _flush() to rewrite all seven files. A single module-level singleton (store) is imported everywhere.

self._users           = {}   # user_id      -> user dict
self._documents       = {}   # doc_id       -> doc dict
self._document_versions = {} # version_id   -> version dict
self._shares          = {}   # share_id     -> share dict
self._share_links     = {}   # token        -> link dict
self._ai_history      = {}   # interaction_id -> interaction dict
self._ai_settings     = {}   # user_id      -> ai provider settings

How it works — the mutating-write pattern

def create_document(self, doc_data: Dict) -> Dict:
    with self._lock:                         # ① acquire the single lock
        self._documents[doc_data["id"]] = doc_data   # ② mutate in-memory
        self._flush()                        # ③ rewrite all seven JSON files
    return doc_data

Three observations, each deliberate:

  1. One lock, not per-entity locks. Simpler reasoning: any write blocks any other write. With small files and few writers, contention is negligible.
  2. Reads are lock-free. Methods like get_document() just return the dict reference. Python's GIL makes single-key dict reads atomic, so a read during a write either sees the old value or the new — never a torn one.
  3. Every write flushes everything. Wasteful at scale, but at PoC scale (megabytes of JSON) it takes milliseconds and guarantees that disk state never lags memory by more than one operation.
⚡ Key Insight — the interface is what matters, not the implementation

Every router in the backend calls store.get_document(...), store.update_document(...), etc. None of them know (or care) that behind this interface is a dict-and-a-lock rather than a SQL query. Swapping this for a real database in production would mean rewriting one file (json_store.py) — every router, service, and test keeps working. This is the "repository pattern" in its simplest form.

🔢 Example — cascading delete

Suppose user Alice deletes a document. The naïve approach would leave orphaned version snapshots, dangling share rows, and unreachable AI-history entries in memory forever. JSONStore.delete_document() does the cascade manually, while holding the lock:

def delete_document(self, doc_id: str) -> bool:
    with self._lock:
        if doc_id not in self._documents: return False
        del self._documents[doc_id]
        for vid in [k for k,v in self._document_versions.items() if v["doc_id"]==doc_id]:
            del self._document_versions[vid]
        for sid in [k for k,v in self._shares.items() if v["doc_id"]==doc_id]:
            del self._shares[sid]
        for tok in [k for k,v in self._share_links.items() if v["doc_id"]==doc_id]:
            del self._share_links[tok]
        for aid in [k for k,v in self._ai_history.items() if v["doc_id"]==doc_id]:
            del self._ai_history[aid]
        self._flush()
    return True

In SQL this would be ON DELETE CASCADE; in wordAI it's five list comprehensions inside one lock.

⚠️ Common Mistake

Don't assume this would survive production traffic. Every write rewrites ~7 files, so writes cost O(total data). Under load, the single lock becomes the bottleneck and the "flush everything" pattern kills disk throughput. That is fine for a PoC serving a few users and a few hundred documents; it is emphatically not fine for 10 000 users.

EasyQ: Why is there a single global lock rather than one lock per entity type?

Because _flush() rewrites all seven files at once. Even a write that only touches _documents needs exclusive access to the disk layer. A per-entity lock would either require fine-grained flushing (one file per lock, more complexity) or would be pointless (you'd need all locks to flush safely anyway).

MediumQ: A reviewer says: "This violates ACID. Writes aren't atomic across files." Is that right?

Strictly, yes — a power failure between _dump_json(users) and _dump_json(documents) could leave the two files inconsistent. The project acknowledges this as a PoC compromise (see DEVIATIONS.md §1). Production would wrap the flush in a single atomic write (e.g. write-temp-then-rename) or use a real database.

We now have a durable place to put users. But how do users prove who they are when they call the API? That's authentication — passwords, JWTs, and a clever trick that makes expired tokens invisible to the user.

3
JWT Authentication with Silent Refresh
Log in once; never see "session expired" while you're editing
🎯 Motivation

Every protected API call needs to answer "who is calling?" instantly, without hitting the database for every single keystroke-save. The classic solution is a short-lived token the client carries on every request. But short-lived means it expires — and if a user is deep in an editing session, a sudden "you were logged out" popup is infuriating. wordAI wants the security of short expirations and the feel of a session that never ends.

💡 Intuition

Think of a theme park with a day pass and a season pass. The day pass (access token) lasts 30 minutes and is scanned at every ride. The season pass (refresh token) lives in your wallet for a week and is only used to get a new day pass when the old one expires. The client never lets the user see the day pass running out — it quietly trades the season pass for a fresh day pass behind the scenes.

📘 Definition — two tokens, two purposes

On login, the server returns a pair of JSON Web Tokens:

TokenLifetimeUsed forClaim type
Access token30 minAttached as Authorization: Bearer … on every protected API call"access"
Refresh token7 daysSent only to POST /api/auth/refresh to get a new access token"refresh"

Both are signed with the backend's JWT_SECRET_KEY using HS256. Both carry the user's id as sub. The server stores nothing — it just verifies the signature and expiry.

How it works — the login path

# backend/app/auth/service.py
def login_user(email: str, password: str) -> Token:
    user = store.get_user_by_email(email)
    if not user or not verify_password(password, user["hashed_password"]):
        raise HTTPException(401, "Incorrect email or password")
    payload = {"sub": user["id"]}
    return Token(
        access_token=create_access_token(payload),    # exp +30 min, type="access"
        refresh_token=create_refresh_token(payload),  # exp +7 days, type="refresh"
    )

Passwords are never stored in clear text — hash_password() uses passlib.CryptContext(schemes=["bcrypt"]), which salts and stretches the password before storage. verify_password() does the constant-time comparison on login.

How it works — protecting a route

# backend/app/middleware/auth.py
oauth2_scheme = OAuth2PasswordBearer(tokenUrl="/api/auth/login")

def get_current_user(token: str = Depends(oauth2_scheme)) -> dict:
    payload = decode_token(token)                       # verifies signature + expiry
    if payload.get("type") != "access":                 # reject refresh tokens here
        raise HTTPException(401, "Expected an access token")
    user = store.get_user(payload["sub"])
    if not user: raise HTTPException(401, "User not found")
    return user

Any FastAPI route that writes current_user: dict = Depends(get_current_user) is now protected. A missing header, an invalid signature, an expired token, or a refresh token sneaked into an access-token slot all produce a 401.

⚡ Key Insight — the silent refresh trick

The magic isn't on the server — it's on the client, in frontend/src/api/axios.ts. An Axios response interceptor watches every reply. If it sees a 401, it:

  1. Marks the original request as "about to retry" (_retry = true) so we don't loop forever.
  2. Calls POST /api/auth/refresh with the stored refresh token.
  3. Stores the new access token, retries the original request with the new header.
  4. Returns the retried response to the caller — as if the 401 never happened.

From the user's point of view, there was a single successful save. From the network tab's point of view, there was a 401, a refresh, and a retry. This is why the DEVIATIONS doc calls it "strictly better UX than a modal."

🔢 Example — a PATCH at minute 31

Alice logged in at 10:00. Her access token expires at 10:30. She keeps typing; at 10:31 useAutoSave fires.

  1. Axios sends PATCH /api/documents/abc with the stale token.
  2. Backend decode_token() raises JWTError → HTTP 401.
  3. Browser's response interceptor catches the 401, calls POST /api/auth/refresh with the refresh token (still valid until 10:00 + 7 days).
  4. Backend returns {access_token: "eyJ..."} — signed with a fresh 30-min expiry.
  5. Interceptor stores it, retries the PATCH. 200 OK.
  6. Alice sees "Saved" — she never knew anything happened.

The isRefreshing flag and failedQueue in axios.ts are there to handle the concurrent case: if five requests fail at once, only one refresh call is made and the other four wait for it.

⚠️ Common Mistake

Don't put the refresh token in a place a protected route can accidentally receive it. get_current_user() explicitly checks type == "access" precisely because a refresh token would otherwise pass signature verification and let an attacker do more than they should. A refresh token is only valid at /api/auth/refresh.

⚠️ Common Mistake

The localStorage-based token store is vulnerable to XSS: any script that runs in the browser can read the token. For a PoC this is acceptable; production should prefer HttpOnly cookies.

Edge case — the WebSocket

WebSockets don't support custom headers in the browser, so the frontend passes the access token as a query parameter: ws://host:8000/ws/{doc_id}?token=eyJ…. The backend (backend/app/websocket/router.py) calls the same decode_token() helper before accepting the upgrade. Share-link guests bypass JWT entirely and pass share_token=… instead — the same router handles both paths.

EasyQ: Why does the access token expire after only 30 minutes if the refresh token lasts a week?

Blast-radius minimisation. The access token is sent on every API call and therefore has many copies in logs, memory, and network paths. A short expiry means a stolen access token is useless in 30 minutes. The refresh token is only used against one endpoint and therefore leaks less — so it can afford a longer lifetime.

MediumQ: What prevents an infinite loop of 401 → refresh → 401 if the refresh token is also expired?

Two guards. First, the interceptor marks the request with _retry = true before retrying, so a second 401 on the same request won't try to refresh again. Second, if the refresh call itself fails, clearAuthAndRedirect() wipes localStorage and navigates to /login, breaking any further retries.

4
Documents, Versions & Sharing
The domain that 80 % of user value flows through
🎯 Motivation

A document editor without shareable, versioned documents isn't a product. Users need four things: create/edit/delete their own docs, restore "oh no, I deleted a paragraph" mistakes, invite specific people, and occasionally paste a link into Slack so anyone with the link can view or edit. wordAI implements all four in a single feature module.

💡 Intuition

Think of a Google Doc. It has one owner, a few named collaborators (each with a role), a revision history you can scroll back through, and a "share this link" button that generates a URL anyone can open. wordAI replicates that mental model almost exactly — just with simpler guts underneath.

📘 Definition — the three access paths to a document

Any HTTP call that touches a document must answer "does this caller have permission?" There are three legitimate answers:

  1. Owner — document.owner_id == current_user.id. Full control.
  2. Explicit share — a row in the shares dict: {doc_id, user_id, role: "editor"|"viewer"}. Created by the owner via the Share Modal.
  3. Share link — a row in the share_links dict: {token, doc_id, role, expires_at}. Anyone presenting the token (logged in or not) gets the role the owner baked into the link.

The helper _get_user_role(doc_id, user_id) returns "owner", "editor", "viewer", or None. Most routes call it once and compare against a minimum-rank requirement: {viewer: 0, editor: 1, owner: 2}.

How it works — the router in one screen

# backend/app/documents/router.py (trimmed)
@router.get("/")                          # list my docs + docs shared with me
@router.post("/")                         # create
@router.get("/{doc_id}")                  # read
@router.patch("/{doc_id}")                # update (also accepts ?share_token=…)
@router.delete("/{doc_id}")               # delete (owner only)

@router.get("/{doc_id}/versions")         # list snapshots
@router.post("/{doc_id}/versions/{vid}/restore")  # roll back

@router.get("/{doc_id}/shares")           # who has access
@router.post("/{doc_id}/shares")          # invite by email/username
@router.patch("/{doc_id}/shares/{uid}")   # change their role
@router.delete("/{doc_id}/shares/{uid}")  # revoke

@router.post("/{doc_id}/share-links")     # create a link
@router.get("/{doc_id}/share-links")      # list active links
@router.delete("/{doc_id}/share-links/{token}")  # revoke a link
@router.get("/via-link/{token}")          # open a doc via link (no login needed)

Notice the PATCH endpoint accepts either a logged-in user or a share_token query parameter. That single polymorphism is why the "anyone with the link can edit" feature is essentially free — all downstream logic is the same.

Version history — snapshot-on-save

Every auto-save triggers a create_version call that stores a complete copy of the HTML content along with a monotonically increasing version_number. Restoring a version is just document.content = version.content + bump updated_at. There is no delta storage, no OT log; a 10 KB document creates a 10 KB version record every time.

⚠️ Common Mistake

If a user types furiously and auto-saves fire every 1.5 s for an hour, the versions list will bloat to ~2400 entries. A production system would coalesce them (e.g. keep per-minute snapshots, or diff-compress). The PoC accepts the bloat — see DEVIATIONS.md §7.

⚡ Key Insight — roles compose down, not up

If Alice is an editor via a direct share and a viewer via a share link, which role wins? The code checks the JWT path first; if the JWT user has any direct role, that's used. Share-link tokens only kick in when no JWT-bound role is found. This is the right default: an authenticated user's permissions are stable and personal; a shared link is incidental.

🔢 Example — creating a link and using it
  1. Alice (owner) calls POST /api/documents/abc/share-links with body {role: "editor", expires_in_days: 7}.
  2. Backend generates a random URL-safe token, stores {token, doc_id: "abc", role: "editor", expires_at, created_by: "alice"}, and returns the token to Alice.
  3. Alice pastes https://wordai.example/shared/xK9p… into Slack.
  4. Bob (no account) clicks it. React routes him to SharedDocPage, which calls GET /api/documents/via-link/xK9p….
  5. Backend's get_doc_via_share_link looks up the token, attaches role "editor" to the response, and Bob's editor mounts in editable mode.
  6. Bob's WebSocket connects with ?share_token=xK9p… instead of a JWT. The WS router sees no user id, validates the share token, and assigns Bob the guest id guest_xK9p…. His edits broadcast to Alice's tab exactly like any other collaborator's.
EasyQ: A user is not the owner of document X, has no direct share row, and presents no share-link token. What does _get_user_role return?

None. The caller then raises HTTP 403 (or 404 if the document genuinely doesn't exist).

MediumQ: The PM asks: "Can we add a third role, commenter, above viewer but below editor?" Where would you change things?

Three places. (1) Add "commenter": 1 to the _rank dict wherever it appears (documents/service, ai/router, websocket/router). (2) Update the Pydantic ShareRole enum in documents/models.py. (3) Decide which endpoints require what rank — e.g. commenting might be gated on rank ≥ commenter, AI on ≥ editor. Business logic sits in service.py; router just dispatches.

Docs get saved. Docs get shared. Now the interesting part: when two people are inside the same doc simultaneously, how do their keystrokes reach each other in ~100 ms?

5
Real-Time Sync Protocol
A WebSocket JSON relay, last-writer-wins, deliberately dumb
🎯 Motivation

Two people are typing in the same document. Without live sync, they'd each see only their own work until someone refreshed — Google Docs would feel like email. With live sync, characters appear in everyone's tab within a heartbeat. The industry-standard way to do this is a CRDT (Conflict-free Replicated Data Type) like Yjs — mathematically elegant, merges divergent edits without conflicts, allows remote cursors. The team tried it and hit walls it couldn't afford to keep debugging. What remained was a question: how simply can real-time sync possibly work?

💡 Intuition

Forget operational transforms and CRDTs. Picture a chat room where everyone occasionally shouts their whole essay. Whoever shouted last wrote the essay everyone now sees. That's it. In a room of two, one person is usually typing at a time, so "last shout wins" looks identical to perfect merging. In a room of twenty simultaneous typists it would be chaotic — but wordAI is not optimising for twenty simultaneous typists.

📘 Definition — the JSON WebSocket protocol

A single WebSocket endpoint (/ws/{doc_id}) accepts two types of client-to-server frames and emits three types of server-to-client frames. All are JSON text.

DirectionTypeShapePurpose
C → Sedit{type, html, origin}Broadcast my local HTML to peers
C → Scursor{type, origin, pos}Broadcast caret position
S → Cedit(same)Peer's edit, for you to apply
S → Ccursor(same)Peer's caret position
S → Cawareness{type, users:[{user_id, username, color}]}Who is currently in the room

The ConnectionManager is stateless about content — it never parses the HTML payload; it only forwards. awareness is derived from its user_info dict and re-broadcast on every connect/disconnect.

How it works — three pieces

① The server is a relay. In backend/app/websocket/manager.py:

async def handle_text_message(self, doc_id, conn_id, raw):
    try: msg = json.loads(raw)
    except Exception: return
    if msg.get("type") not in ("edit", "cursor"): return
    await self.broadcast_text(doc_id, raw, exclude_conn_id=conn_id)

That is the entire sync logic on the server. Receive a frame → forward it to everyone else in the same doc_id room. Nothing is stored. Nothing is transformed. If a frame is malformed, it is silently dropped.

② Presence is derived, not broadcast-by-clients. On connect(), the server assigns the new socket a colour from an 8-colour palette (#F44336, #2196F3, …) and stores {user_id, username, color} in user_info[doc_id][conn_id]. It then emits an awareness frame to the whole room. The PresenceBar on the frontend just renders the users list from the last awareness it received.

③ The client debounces sends and applies receives carefully. In frontend/src/hooks/useRealtimeSync.ts:

const THROTTLE_MS = 80;
const onUpdate = () => {
  if (applyingRemoteRef.current) return;          // don't echo remote edits
  const elapsed = Date.now() - lastSentAtRef.current;
  if (elapsed >= THROTTLE_MS) flush();             // leading edge: send NOW
  else if (!sendTimerRef.current)
    sendTimerRef.current = setTimeout(flush, THROTTLE_MS - elapsed); // trailing
};

The throttle is leading-edge: the first keystroke in a burst fires immediately (no warm-up feel), and subsequent keystrokes within 80 ms coalesce into a single trailing send with the latest HTML. On receive:

if (msg.type !== 'edit' || msg.origin === clientId) return;      // ignore own echoes
const { from, to } = editor.state.selection;                     // save caret
applyingRemoteRef.current = true;
editor.commands.setContent(msg.html, false);                     // replace content
editor.commands.setTextSelection({ from: Math.min(from, size), to: Math.min(to, size) });
applyingRemoteRef.current = false;

applyingRemoteRef is the guard that prevents the echo chamber — when setContent fires the editor's own update event, onUpdate bails out instead of re-broadcasting what we just received.

⚡ Key Insight — the caret preservation trick

Replacing the document's HTML normally nukes the user's caret, producing the "cursor jumps to the top" bug that plagues naïve live-sync implementations. The fix is 3 lines: remember selection.from and to before setContent, clamp them to the new document size, then setTextSelection. Because both documents are nearly identical (the peer only typed one character), the clamped offsets land on the right visual position.

🔢 Example — a typing burst from Alice, as seen by the server

Alice types "hello" at t = 0 ms. What does the server see?

t (ms)Event in Alice's browserFrame sent?
0Types "h"✅ leading edge — sends {edit, "<p>h</p>", alice-a9f2}
40Types "e"⏱ schedules trailing (in 40 ms)
70Types "l"⏱ still scheduled; timer will grab latest HTML
80(timer fires)✅ sends {edit, "<p>hel</p>", alice-a9f2}
100Types "l"⏱ schedules trailing
140Types "o"⏱ updates scheduled HTML
160(timer fires)✅ sends {edit, "<p>hello</p>", alice-a9f2}

Five keystrokes → three frames. Bob sees the content visibly growing, but without the bandwidth cost of one frame per key.

⚠️ Common Mistake — misunderstanding last-writer-wins

If Alice and Bob type into the same paragraph at the same time, the later frame wins. Alice types "cat", Bob types "dog" into the same spot; whichever arrives at the server second is the one everyone sees. No merging, no operational transform. This is fine for the typical Google-Docs-ish workflow (one person types at a time with occasional interleaving) and is the baseline the assignment explicitly allows. It is NOT a CRDT. The project acknowledges this trade-off in DEVIATIONS.md §2.

⚠️ Common Mistake

Viewers (users with viewer role, so editor.isEditable === false) don't send, but they do receive. The frontend gates sending with if (!editor.isEditable) return;. Skip that check and a read-only user would try to broadcast, get rejected by the role check, and flood console errors.

EasyQ: Why does every frame carry an origin field?

Because the server fan-outs to "all other sockets" including the sender's other tabs. The origin string is a random client id generated on first mount; the receive handler ignores any frame whose origin matches the local client, so your own edits never round-trip back to overwrite your caret.

MediumQ: If the backend crashes and restarts, what happens to the document content? What happens to presence?

Content survives — it was durably written to backend/data/documents.json by the HTTP auto-save path. When clients reconnect, they reload the document from the API and resume.
Presence resets — the user_info dict lived only in memory. The new ConnectionManager starts empty and repopulates as clients reconnect. Users briefly see each other disappear and reappear in the presence bar.

HardQ: Why not also persist WebSocket edits on the server, to make the HTTP autosave redundant?

You could, but you'd then have two write paths racing for the same file. The current design deliberately keeps the WebSocket layer stateless: one owner of writes (HTTP), one owner of fan-out (WebSocket). Simpler to reason about, easier to test, and the ~1.5 s save-debounce is fast enough for a PoC. A production system might well invert this — persist via WebSocket on the server and treat HTTP save as a fallback.

Collaboration is solved. Now meet the most marketable feature: the AI panel. How does a single "Rewrite" button drive four different LLM providers, stream tokens in real time, and record every interaction?

6
AI Multi-Provider Abstraction
One button, four LLM providers, token-by-token streaming
🎯 Motivation

Assignment 1 mandated "changing the LLM provider should touch one place." In 2026, users also bring their own keys — some prefer DeepSeek for cost, others OpenAI for familiarity, a few run a self-hosted OpenAI-compatible endpoint. The AI module has to (a) abstract the provider, (b) let each user choose theirs and supply their own key, (c) stream tokens back to the browser so the UI feels alive, and (d) remember what the AI said so we can show history and audit costs.

💡 Intuition

Picture a translator at a UN podium. The diplomat (user) points at a feature ("Rewrite this in a formal tone"). The translator looks up the diplomat's preferred interpreter from their badge (per-user settings), hands that interpreter the appropriate cue card (a prompt template), and relays the interpreter's output word by word to everyone in the hall (SSE streaming). The translator doesn't care whether the interpreter speaks with a French or Japanese accent — they all produce words that reach the audience.

📘 Definition — the four moving parts
  1. Feature enum (AIFeature) — one of rewrite, summarize, translate, expand, grammar, custom, autocomplete.
  2. Prompt templates — plain-text files in backend/app/ai/prompts/*.txt, one per feature, with {placeholders} for dynamic values.
  3. Provider resolver (_get_client_config) — per-user lookup that returns (api_key, base_url, model, provider). Falls back to an env-level DeepSeek key if the user hasn't configured one.
  4. Stream generator (stream_ai_response) — an async generator that opens a streaming call to the chosen provider and yields SSE frames until the provider finishes.

How it works — the three-step flow

Step 1. Build the prompt. _build_prompt() maps the feature to a template file and fills in placeholders:

if feature == AIFeature.rewrite:
    return _prompt_loader.render(
        "rewrite",              # loads prompts/rewrite.txt
        text=text,
        tone=opts.get("tone", "neutral"),
        context=context,
    )

The template itself is plain English:

You are an expert writing assistant. Rewrite the following text with a {tone}
tone. Preserve the meaning and key information. Output only the rewritten
text, nothing else.

Original text:
{text}

Context (surrounding text for reference):
{context}

Prompts are cached on first read. Tweaking a prompt is a file edit — no code change, no redeploy of logic.

Step 2. Resolve the provider and open a stream.

api_key, base_url, model, provider = _get_client_config(user_id)

if provider == "claude":
    async with anthropic.AsyncAnthropic(api_key=api_key).messages.stream(
        model=model, max_tokens=2048,
        messages=[{"role": "user", "content": prompt}]
    ) as stream:
        async for text in stream.text_stream: ...
elif provider == "deepseek":
    _deepseek.configure(api_key=api_key, base_url=base_url, model=model)
    stream = await _deepseek.stream(prompt)
    async for chunk in stream: ...
else:  # openai, grok, custom openai-compatible
    openai_client = AsyncOpenAI(api_key=api_key, base_url=base_url)
    stream = await openai_client.chat.completions.create(
        model=model, messages=[{"role":"user","content":prompt}], stream=True)
    async for chunk in stream: ...

Three branches, one contract: each yields text chunks as they arrive. Claude uses the Anthropic SDK's message stream; everyone else uses the OpenAI SDK (which DeepSeek, Grok, and custom endpoints all speak natively).

Step 3. Yield SSE frames and log the interaction.

for each chunk:
    full_response.append(delta)
    yield f"data: {json.dumps({'chunk': delta, 'done': False})}\n\n"

# when provider finishes:
_log_interaction(interaction_id, doc_id, user_id, feature,
                 input_text, prompt_used, response_text)
yield f"data: {json.dumps({'chunk': '', 'done': True, 'interaction_id': interaction_id})}\n\n"

Each frame is one SSE event. The frontend consumes them with parseSSEStream in frontend/src/api/ai.ts and appends chunk to React state until done: true.

⚡ Key Insight — the two axes of abstraction

There are actually two abstractions stacked here:

  • Feature abstraction — adding a new feature means adding one .txt file and one elif branch in _build_prompt. No change to the provider code.
  • Provider abstraction — adding a new provider means adding one elif branch in stream_ai_response. No change to any prompt or feature.

The 7 × 4 matrix of (feature × provider) is implemented as 7 + 4 additions, not 28 — because the prompt and provider axes are orthogonal.

Authentication — the resolver accepts either a JWT or a share-link token

AI calls come through /api/ai/{doc_id}/stream. The route calls _resolve_actor(request, doc_id, share_token) which tries JWT first, then share-token. Either path must resolve to a caller with role ≥ editor (viewers cannot burn the owner's API credits) — otherwise 403. Only after resolution does the stream begin, so unauthenticated clients get an immediate error, not a partial response.

🔢 Example — user clicks "Rewrite in formal tone"
  1. Frontend useAI.streamAI() posts {feature:"rewrite", selected_text:"hey team lets ship", options:{tone:"formal"}} to /api/ai/abc/stream.
  2. Backend resolves caller (JWT → user Alice, role=owner), builds prompt by filling rewrite.txt with the text and tone.
  3. Alice's AI settings say provider: "openai", model: "gpt-4o-mini". Backend opens a streaming chat-completions call.
  4. OpenAI yields chunks: "Dear", " team,", " I", " would", …
  5. Each chunk becomes one SSE frame: data: {"chunk":"Dear","done":false}\n\n, then data: {"chunk":" team,","done":false}\n\n, …
  6. Frontend appends to result state; the AI Result Card visibly types the rewrite.
  7. When OpenAI is done, backend logs the interaction and yields a final {chunk:"",done:true,interaction_id:"i-42"} frame.
  8. User clicks "Accept" → frontend posts to /interactions/i-42/outcome with accepted:true. Backend updates the history row so analytics know which suggestions users actually took.
⚠️ Common Mistake — keys in the browser

The user supplies their own API key via the AI Settings modal. That key is stored on the backend (ai_settings.json) keyed by user id, not in the browser. Never expose an API key to the frontend — a curl against wordAI's backend uses the user's JWT and the backend looks up the key. This is also why AI endpoints require an editor role: a viewer who could call /api/ai/… would effectively drain the owner's LLM credits.

⚠️ Common Mistake — SSE is not JSON

Each SSE event is the literal string "data: " + json_text + "\n\n". The \n\n terminator tells the client "event boundary." Forget it and the browser's EventSource parser never emits a message. The backend uses f"data: {json.dumps(...)}\n\n" — that exact format.

EasyQ: Why is ghost-text autocomplete just another feature enum value instead of its own endpoint?

Because the underlying protocol (prompt → provider → SSE) is identical. Only the prompt template (autocomplete.txt) and the input shape (it uses context_before / context_after instead of selected_text) differ. Reusing /api/ai/{doc_id}/stream kept the router and frontend plumbing tiny.

MediumQ: Imagine Anthropic ships a brand-new provider tomorrow. What's the minimum change to support it?

If it's OpenAI-API-compatible (most new providers are): add it as an option in the AI Settings UI; set base_url to the new provider's endpoint; no backend code changes at all. If it has a unique SDK (like Anthropic): add one elif in stream_ai_response that uses the new SDK to yield text chunks — about 10 lines. Everything else — prompts, UI, history — is unchanged.

We've covered how AI results get to the browser. But what shows them? What actually holds the user's text? That's Tiptap — and tucked inside Tiptap is the custom extension that powers Copilot-style ghost text.

7
Tiptap Editor & Ghost-Text Autocomplete
A rich-text core, plus one custom plugin that mimics Copilot
🎯 Motivation

A document editor needs to handle bold, italics, headings, lists, tables, links, images, code blocks, and everything in between. Writing that from scratch is a year of work. Tiptap wraps ProseMirror (a well-engineered contentEditable abstraction) and gives wordAI the whole thing for free. On top of that, the product wanted Copilot-style ghost text — faint grey predictions that appear after you pause and disappear when you keep typing. That required one new Tiptap extension.

💡 Intuition

Tiptap is the editor's body (it holds the document and dispatches events). The ghost-text extension is a small whisper in the editor's ear: "draw this grey text at the caret but don't treat it as real content." If the user accepts (Tab), the whisper becomes real characters. If they keep typing, the whisper vanishes. The user never sees the bookkeeping.

📘 Definition — Tiptap in wordAI

The editor is constructed in RichTextEditor.tsx with a starter kit (paragraphs, headings, lists, bold, italic, etc.) plus extensions for tables, task lists, links, images, code blocks with syntax highlighting (lowlight), character count, placeholder, typography, text alignment, underline, colour, and the custom GhostTextExtension. The editor instance is exposed via an EditorContext so the toolbar, AI panel, and floating AI menu can all act on the same document.

How it works — the three hooks that drive the editor

HookTriggerWhat it does
useRealtimeSyncEvery editor.updateThrottles 80 ms, sends JSON edit frame via WebSocket. Receives peers' edits and applies them.
useAutoSaveContent changesDebounces 1.5 s, PATCHes /api/documents/{id}. Flushes on pagehide using fetch(..., {keepalive: true}).
useGhostText3 s of typing inactivityStreams feature: "autocomplete" from the AI endpoint. Renders the result as ProseMirror decorations (grey inline text, not real content).

The ghost-text plugin — how the grey text works

ProseMirror has a concept called decorations: visual overlays that render as if they were content but don't affect the document model. GhostTextExtension.ts registers a ProseMirror plugin that holds a small state object {text, isPredicting} and produces a widget decoration at the caret position whenever text is non-empty.

The hook in useGhostText.ts orchestrates:

  1. On every editor.update, check: is there a selection? is the caret at end of doc? has 3 s of inactivity elapsed?
  2. If yes, cancel any previous stream, start a new SSE call with context_before (everything before caret) and context_after (everything after).
  3. For each incoming chunk, call editor.commands.setGhostText(accumulated, isPredicting=true) — the plugin updates its decoration.
  4. When the stream ends, set isPredicting=false (the plugin now shows the ghost as "ready to accept").
  5. Tab accepts → the plugin inserts the text as real content and clears itself. Esc or any further keystroke dismisses.
⚡ Key Insight — why decorations, not real content

If the ghost text were real document content, every WebSocket peer would see the ghost prediction appear in their document — because the real-time sync broadcasts HTML. Decorations are purely client-side visual sugar: ProseMirror doesn't include them in editor.getHTML(), so they never leak into the edit frame. Each user sees their own ghost text and nobody else's. This is the same mechanism Copilot uses in VS Code.

🔢 Example — ghost text appearing and being accepted
  1. Alice types "The quarterly revenue was " then stops.
  2. 3 s pass with no keystrokes. useGhostText fires.
  3. It grabs context_before = "The quarterly revenue was ", context_after = "".
  4. Streams from the AI provider using autocomplete.txt.
  5. Provider yields "up", " 12%", " year", "-over", "-year", …
  6. Each chunk updates the plugin's ghost-text state. Alice sees "up 12% year-over-year…" appear in grey.
  7. Alice presses Tab. The plugin inserts the accumulated ghost as real content, clears itself, WebSocket broadcasts the new full HTML to peers.
  8. Bob's tab sees the completed sentence appear all at once, as if Alice had typed it in one burst.
⚠️ Common Mistake

Ghost text must be cancelled the moment the user types, clicks elsewhere, or selects text. The hook has three cancellation paths: handleUpdate (on any edit), handleSelectionUpdate (on any selection change), and the dismissedRef (set by Escape, cleared by next keystroke). Miss any of them and the ghost will "fight" the user's typing, replacing characters they already entered.

EasyQ: Why is ghost text disabled when the caret is not at the end of the document (textAfter.trim().length > 0)?

Because ghost text is a continuation, not an insertion. Predicting what should follow when there's already content after the caret would usually produce gibberish (ghost text ≠ the existing next paragraph). Restricting to end-of-doc gives the model a clean "write the next few words" task.

MediumQ: Why does the hook prepend a leading space to the ghost text when the text before the caret doesn't end with whitespace?

The LLM doesn't know whether the user already typed a trailing space. If the user stopped at "hello" (no trailing space) and the model returns "world", directly inserting it yields "helloworld". The needsLeadingSpace check in useGhostText prepends " " so the display becomes "hello world".

8
Engineering Deviations — what changed and why
The list of trade-offs that define wordAI's personality
🎯 Motivation

Assignment 1 was an architecture document. Assignment 2 was the implementation. The gap between them is where engineering judgement lives. As the PM, you should be able to defend every deviation to a grader (or a future hire): was it a compromise or an improvement? What did it cost? What did it buy? DEVIATIONS.md is the primary artefact here — this unit distils its eight entries into a PM-readable form.

💡 Intuition

Think of Assignment 1 as the architect's blueprint and Assignment 2 as the house that actually got built. Some rooms came out exactly as drawn. Some were shrunk to fit the budget. A couple became bigger than planned. None of the deviations are surprises — each has a paper trail explaining the substitution. That discipline is how you ship in a constrained timeline without creating a documentation mess later.

📘 Definition — the eight deviations
#What changedVerdict
1PostgreSQL → JSON files on diskImprovement for PoC scope
2CRDT (Yjs) → simple JSON WebSocket broadcast, last-writer-winsCompromise on conflicts, improvement on reliability
3Single provider (OpenAI) → multi-provider abstractionImprovement
4Added ghost-text autocomplete (not in A1)Improvement (bonus feature)
5Added share-by-link (not in A1)Improvement (bonus points)

7OT log for versions → full snapshotsCompromise for simplicity

How to think about each deviation

#1 JSON over PostgreSQL. The assignment permits file-based persistence for a PoC. Trading SQL for JSONStore removes setup friction (no Docker, no migrations) and costs only production-grade durability — which the PoC doesn't need. The repository pattern means swapping back is a one-file change.

#2 JSON broadcast over Yjs (CRDT). An earlier implementation used Tiptap's Collaboration extension with y-websocket. Local edits silently failed to push — no sync-update frames ever reached the server. After several hours at the Yjs/y-websocket/Tiptap boundary the team chose a protocol they could fully reason about. The cost is the CRDT bonus (conflict-free concurrent edits) and remote cursors; the gain is reliability and debuggability. This is the most consequential deviation.

#3 Multi-provider LLM. A1 specified OpenAI-only. The team generalised to DeepSeek, OpenAI, Anthropic, Grok, and any OpenAI-compatible endpoint — more ambitious than the assignment, and strictly better. Users bring their own keys via a settings modal, so the project itself doesn't have to budget API spend.

#4 Ghost-text autocomplete. Not in A1. Added because the AI streaming infrastructure was already there and the feature is table stakes for a 2026 AI editor (Copilot, Notion AI). Pure win.

#5 Share-by-link. Not in A1. Added because it reuses the existing role-check code and earns bonus points. The "anyone with the link" pattern is what makes Google Docs shareable in Slack.

#6 AI history as JSON log. Follows mechanically from #1. The schema is identical to what A1 specified; only the storage layer differs.

#7 Version snapshots over OT log. Full snapshots are simpler to implement, easier to reason about on restore, and acceptable storage overhead for small documents. An OT log (operational transform) would be delta-based and space-efficient but dramatically more complex. Production would revisit; PoC skips.

#8 Silent token refresh. A1 suggested a "session expired" modal. The team chose an Axios interceptor that refreshes invisibly, because breaking a collaborative editing session on token expiry is a terrible UX. Strictly better.

⚡ Key Insight — the pattern behind every deviation

Six of the eight deviations share one template: "replace the sophisticated solution with the simpler one that still satisfies the requirement." JSON files instead of Postgres, snapshots instead of OT logs, broadcast instead of CRDT, one SSE endpoint per feature instead of seven. This is the PoC's dominant engineering principle: deliver the contract, not the implementation ambition. The remaining two (#3, #4, #5, #8) are upgrades — the team spent the saved complexity budget on features with higher marginal user value.

🔢 Example — how to answer a grader's challenge

Grader: "Why didn't you use Yjs? The CRDT bonus was worth points."
You: "We tried. The Yjs + y-websocket + Tiptap integration was silently dropping local edits in our setup; we burned several hours at that boundary without resolution. We chose to ship a protocol we fully understood (JSON broadcast + last-write-wins) rather than one we couldn't confidently reason about. The assignment explicitly allows last-write-wins as the baseline; we gave up the CRDT bonus in exchange for a reliably-working core feature and documented it in DEVIATIONS.md §2."

⚠️ Common Mistake

Don't present the deviations as failures. Each one is an engineered trade-off: a written argument that a cheaper substitution satisfies the contract. Pair every "we dropped X" with "we gained Y" — that's what turns DEVIATIONS.md into a positive artefact instead of a confession.

MediumQ: Which deviation cost the project bonus points? Which earned bonus points?

Cost: #2 (CRDT dropped). Earned: #5 (share-by-link), plus partial-acceptance of AI suggestions, plus Playwright E2E tests (mentioned in the README).

HardQ: Suppose an investor says "productise this in 3 months." Which deviation would you reverse first, and why?

#1 (JSON → PostgreSQL). It's the lowest-risk upgrade (the repository interface is already there), it unlocks multi-instance deployment (no single-process file lock), and it's a prerequisite for every other production concern: backups, replicas, analytics, migrations. #2 (CRDT) is a distant second — worth doing before you have concurrent editors, but irrelevant if you're still single-instance and users mostly take turns.

Synthesis — tying it all together

wordAI is a single coherent story about separating concerns by transport. HTTP carries durability (CRUD, auth, auto-save). WebSocket carries liveness (edit broadcast, presence). SSE carries incremental output (AI streaming). Everything else is the minimum plumbing to connect the pieces.

Follow one user action end-to-end and you see the whole architecture: a keystroke drives three concurrent side-effects (WebSocket broadcast, auto-save HTTP, ghost-text SSE after 3 s), each owned by a different hook, each hitting a different backend entry point, each flowing to a different destination (peers, disk, LLM). The fact that they feel like one smooth editor to the user is the system's whole trick.

⚡ In three sentences

wordAI is a React + FastAPI document editor where CRUD is HTTP, live edits are a WebSocket JSON relay with last-writer-wins, and AI features stream tokens via SSE across an abstraction that plugs into four LLM providers. All data is persisted to seven JSON files via a lock-guarded in-memory store, and user sessions never visibly expire because an Axios interceptor silently refreshes JWTs on every 401. The project is a PoC that deliberately trades CRDT conflict-resolution and PostgreSQL durability for reliability, debuggability, and ship-speed.

Self-Test — 10 questions

Easy1. Name each of the three transports wordAI uses and one feature each is responsible for.

HTTP — auth / document CRUD / auto-save. WebSocket — live edit broadcast / presence. SSE — AI feature streaming / ghost-text autocomplete.

Easy2. What is the name of the backend singleton that holds all persistent data in memory?

store, an instance of JSONStore from backend/app/storage/json_store.py. Every router imports it; every mutation goes through it.

Medium3. Explain in your own words why the backend's WebSocket manager deliberately does NOT parse the HTML payload of edit frames.

It's a pure relay. Parsing HTML would (a) couple the sync layer to the editor's output format, (b) add CPU cost per frame, and (c) duplicate state that's already owned by the HTTP persistence path. Keeping it opaque means we can change the edit schema (e.g. to ProseMirror JSON) without touching the server.

Medium4. A user reports: "After I leave my laptop for an hour and come back, my next save fails." What's the likely cause, and what's the actual code path that usually prevents this from being visible?

Cause: the 30-min access token expired. Normally, the Axios response interceptor catches the 401, calls POST /api/auth/refresh, and retries the save — the user sees nothing. If the user reports a failure, either (a) the refresh token also expired (7 days), or (b) the refresh endpoint itself returned an error. Either way clearAuthAndRedirect() fires and they're pushed to /login.

Medium5. Why is ghost-text autocomplete implemented as ProseMirror decorations rather than real document content?

Because decorations are client-side only — they don't appear in editor.getHTML(). If ghost text were real content, every WebSocket peer would see the grey prediction in their own document (and worse, if two people were editing, their predictions would race). Decorations make the prediction purely a local UX affordance.

Medium6. The product manager wants to add a seventh AI feature: "Tone analysis — tell me if this paragraph sounds hostile." Walk through every place you'd touch.

(1) Add tone_analysis to AIFeature enum in backend/app/ai/models.py. (2) Create backend/app/ai/prompts/tone_analysis.txt. (3) Add an elif branch in _build_prompt. (4) Add a button to frontend/src/components/ai/AIPanel.tsx and a feature-icon mapping. (5) Optionally update FeatureButton to surface the new option. Provider code, routing, storage, auth: zero changes.

Hard7. Two users, Alice and Bob, open the same document. Alice's access token expires mid-session. She keeps typing. Bob sees her edits continue without a pause. Explain why.

Two reasons. (a) The live-edit path is WebSocket, and the WebSocket was authenticated once at connect time with whatever token was valid then. The backend doesn't re-verify the JWT on every frame; as long as the socket stays open, frames flow. (b) The auto-save HTTP path independently refreshes the JWT via the Axios interceptor on 401. So Alice's WebSocket stays alive, her saves silently refresh, and Bob sees a smooth stream of edits.

Hard8. Compare the share mechanism and the share-link mechanism: when would a PM pick one over the other for a new feature (say, a classroom-wide document)?

Direct shares are tied to a specific user-id (needs an account) and are individually revocable; use when you want per-person accountability. Share links are opaque tokens (no account required) and are revoked by invalidating the token; use when you want zero-friction distribution (paste in Slack / email). For a classroom where you know the 20 students' emails, direct shares give each student their own revocable grant. For a 300-student lecture where you don't maintain the roster, a single editor link is simpler.

Hard9. Why does the AI endpoint require role ≥ editor rather than just any authenticated caller?

Because AI calls consume the owner's API key (or their own if configured). A viewer who could call /api/ai/... could drain credits by repeatedly asking for long completions. The editor-or-above gate is the abuse-prevention layer. It also semantically matches user expectations: viewers read, editors write, AI is a form of writing.

Hard10. If you had one engineer-week to productionise wordAI, which deviation would you reverse and which order of tasks would you attempt?

Reverse #1 (JSON → PostgreSQL) because it's the bottleneck for every further production concern: multi-instance deployment, backups, analytics, migrations. Order: (1) Swap JSONStore implementation for a SQLAlchemy-backed one (routers unchanged). (2) Add connection pooling and a migrations tool (Alembic). (3) Move API keys out of JSON and into an encrypted secrets table. (4) Rate-limit /api/ai/* per user. (5) Introduce a WebSocket server that persists (so scaling to 2+ backend processes doesn't lose presence). CRDT (#2) is deferred; it's only worth doing when the product has concurrent-editor pain.

Exercise 1Trace an auto-save from keystroke to disk

Given: A logged-in user types a character in the editor at 14:02:30.000.

Find: The full list of events that lead to the character being durably written to documents.json.

Solution:

  1. 14:02:30.000 — Tiptap's update event fires. Both useRealtimeSync.onUpdate and useAutoSave.useEffect run.
  2. 14:02:30.002 — useRealtimeSync (leading edge) sends a {type:"edit"} WebSocket frame. Peers see the change within ~100 ms. No disk impact.
  3. 14:02:30.002 — useAutoSave marks status "unsaved" and schedules save(content) for 14:02:31.502 (1500 ms later). Any keystroke before that time resets the timer.
  4. 14:02:31.502 — Timer fires (assuming no further typing). save() calls documentsApi.update(docId, {content}).
  5. 14:02:31.503 — Axios attaches Authorization: Bearer <access_token> and sends PATCH /api/documents/abc.
  6. 14:02:31.510 — FastAPI router update_doc runs, resolves get_optional_user → user. Calls update_document(doc_id, user_id, body) in documents/service.py.
  7. 14:02:31.511 — Service checks role ≥ editor, creates a version snapshot via store.create_version(), then calls store.update_document().
  8. 14:02:31.512 — JSONStore acquires the lock, updates _documents[abc], calls _flush() which rewrites all seven JSON files.
  9. 14:02:31.515 — Router returns the updated DocumentResponse. Frontend sets saveStatus: "saved". User sees the "Saved" indicator.

Result: ~1.5 s from keystroke to durable write; ~100 ms from keystroke to peer visibility. The two paths are completely independent.

Exercise 2Reason about the silent-refresh guarantees

Given: A user's access token expired five minutes ago; their refresh token is still valid. They trigger three parallel API calls (save, fetch versions, fetch share list).

Find: What exactly happens in the Axios interceptor? How many refresh calls are made?

Solution:

  1. All three requests go out with the stale access token, nearly simultaneously.
  2. The backend rejects all three with 401. The responses trickle back into the interceptor.
  3. The first 401 sets isRefreshing = true, marks its request _retry = true, and calls POST /api/auth/refresh.
  4. While that call is in flight, the second and third 401 arrive. The interceptor sees isRefreshing === true and pushes their (resolve, reject) functions into failedQueue — these promises are pending.
  5. The refresh call succeeds. processQueue(null, new_access_token) resolves both pending promises with the new token.
  6. Each queued promise's .then() updates its request's Authorization header and re-sends via api(originalRequest).
  7. All three original calls eventually succeed and return to their callers, as if there had been no 401.

Result: Exactly one refresh call is made even though three 401s were received. This is the purpose of the isRefreshing flag — it serialises refreshes and batches the waiters.

Exercise 4Explain what happens when two users type into the same paragraph simultaneously

Given: Alice and Bob both have the same paragraph selected. At t = 0 Alice types "cat" into a specific spot; at t = 10 ms Bob types "dog" into the same spot.

Find: What does each user see, and what is eventually persisted?

Solution:

  1. At t = 0, Alice's onUpdate throttle fires (leading edge) — sends {edit, "<p>cat...</p>", alice}.
  2. At t = 10, Bob's onUpdate throttle fires — sends {edit, "<p>dog...</p>", bob}.
  3. Server receives Alice's frame at t ≈ 2, fans it out: Bob's tab applies "cat" (and loses "dog" briefly, depending on timing).
  4. Server receives Bob's frame at t ≈ 12, fans it out: Alice's tab applies "dog".
  5. Final state in both tabs: whichever frame arrived at the server second. If Bob's arrived second, both users end up seeing "dog".
  6. Meanwhile, Alice's auto-save debounce fires ~1.5 s later with whatever content Alice's editor now holds — "dog" in this example, because she received Bob's frame.
  7. The document in documents.json therefore reflects "dog" — Alice's "cat" is lost.

Lesson: This is the last-writer-wins contract. It is fine for serialised typing (which is the overwhelmingly common case). It fails under true simultaneous conflict. The project acknowledges this explicitly; a CRDT would resolve it. A PM should communicate this limit to sales/marketing — don't promise "conflict-free concurrent editing" in copy.

Exercise 5Defend the architecture to a skeptical CTO

Given: A CTO reviews the codebase and says: "JSON files? A WebSocket relay with no conflict resolution? This is a toy."

Find: Write a two-paragraph response that is honest, precise, and defends the choices.

Solution:

"You're right that this wouldn't run a production SaaS. It isn't meant to — it's the Proof-of-Concept deliverable for AI1220 A2, with a specific brief that allows file-based persistence and last-writer-wins conflict semantics. The architectural contract with A1 is honoured at the interface level: every router and service talks to a JSONStore that mirrors the relational schema one-for-one. Swapping it for SQLAlchemy over Postgres is a single-file change. Every deviation from A1 is documented in DEVIATIONS.md with the rationale and a verdict (compromise vs. improvement).

Where we went beyond the brief is also real: the AI layer is multi-provider (DeepSeek / OpenAI / Anthropic / Grok / custom OpenAI-compatible) with per-user key storage; we added Copilot-style ghost-text autocomplete; we built share-by-link with configurable roles on top of the direct-share mechanism; the session never visibly expires because Axios silently refreshes JWTs. The CRDT is the one substantial compromise — we tried Yjs, it silently lost frames at the Tiptap integration boundary, and we judged shipping a reliably-working JSON broadcast more valuable than debugging a black-box CRDT stack under time pressure. For a PoC that a grader can run with one pip install and one npm install, we think the trade-offs are defensible."

The structure of a good defence: (1) acknowledge the scope, (2) point at the interface (not the implementation) as the architecture, (3) name one upgrade and one compromise, (4) reference the written artefact.
